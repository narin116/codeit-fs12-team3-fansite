## 느낀점
**김나린**<br/>
 이전 팀프로젝트에서 얻은 경험을 바탕으로.. push하기 전 **pull하는 게 모두 습관**이 된 것 같다. 그래서인지 전 프로젝트보다 **충돌이 나는 경우가 훨씬 적었던 것 같다.** 내 개인 브랜치를 가지고 PR을 하는 방식도, 너무 모든 커밋을 dev에 기록에 남기는 것보다, 그래도 중요한 것들 위주로 굵직하게 dev에 기록을 남길 수 있으니 더 깔끔한 것 같아 좋았다.
<br/><br/>

**김상우**<br/>
**Git 과 Github** 를 같이 사용함으로써 협업을 하면서 **소통이 원활하게 돌아가야 한다는 것**을 크게 느꼈습니다.
혼자 작업할 땐 수시로 pull 받지 않고도 가능할 것 같지만 협업을 하면서 **pull을 자주 땡겨오는 습관**이 들어서 좋은 습관을 들인 것 같습니다. 그리고 협업을 하는 중이라서 서로 오류 나지 않으려고 소통을 계속 하여 push 하고 pull을 했던 것 같습니다.
개인 브랜치를 만들어서 바로 main 으로 가지않고 dev 에 거쳐 수정을 하고 오류 수정도 하여 끝이 나면 마지막에 main으로 pr 하는 새로운 방법을 알게 되어서 좋았습니다.
<br/><br/>

**김서하**<br/>
 혼자 작업 할 때와 달리, git과 github를 활용하니 신경 쓸 것이 더 생겨서 어렵게 다가왔던 것 같다.
내가 작업 하다 가도 pull 하는게 습관 되고 오류도 경험하니 어느정도 대처법도 알게 돼서 많은 도움이 된 것 같다. 확실히 **github**를 사용하니 **작업의 효율성**이 높아진 것을 느꼈고, 경험이 더 쌓여서 나중에는 github 사용을 두려워하지 않게 됐으면 좋겠다.
오류가 나지 않기 위해서는 **(특히 push전이나 pr전에 !)pull을 하는 습관**도 좋고, 작업 하기 전 **내가 어느 브랜치에 있는지** 한번 더 체크하는 습관도 좋은 것 같다. 제일 중요한 건 **팀원들과의 원활한 소통**이라는 것을 다시 한번 느끼는 시간이였다.
<br/><br/>

**박태진**<br/>
 Git & Github : Conflict 가 있을 때, 사용자에게 선택권을 주는데, 상대방이 언제 어떤 코드를 썼는지 **커밋을 열심히 안 읽으면** Current Change와 Incomming Change, 둘 다 남기기 중에 어떤 걸 골라야할지 고민이 많이된다. 팀 활동을 할 때, **PR에 컨펌권한을 부여해서 (정해진 시간에) 확인된 것만 DEV로 Merge하는 것이 좋다고 생각했다.**(권한을 가진 사람이 고생하겠지만) 왜냐하면 내가 수정 후에 Dev 를 Pull 했을 때 내 기능 브렌치랑 Conflict이 떠서 고민되는 경우가 종종 있었기 때문이다. 협업 간에 ****코드 리뷰나 PR시간을 미리 정해서**** 일정에 맞출 수 있어서 좋았다.<br/>
HTML & CSS: 구현하고 싶은 기능이 있으면 Ai Chatbot에게 물어보았다. 애초에 모르는 기능들이 대부분이라서 **원리를 설명**해달라고 했는데, 좋은 학습법인것 같다.
<br/><br/>

**유지민**<br/>
 **common.css**나 **연결 주소 경로 등 문제점**이 생기는 걸 보면서
생각했던 것보다 훨씬 다양한 장치가 웹 페이지 안에서 작동되고 있음을 체감했다.
또 팀 프로젝트를 통해 **PR 이나 공통 파일들이 효율성을 높인다**는 걸 이해할 수 있는 시간이었다.
<br/><br/>

**최광헌**<br/>
 GitHub를 활용하면서 **협업의 중요성**을 깊이 느꼈다.
브랜치를 분리하고 Pull Request를 통해 코드를 병합하는 과정에서 **서로의 코드를 이해하고 피드백을 주고받는 경험**이 인상 깊었다.
특히 **merge** 과정에서 **충돌**이 발생했을 때, **소통**이 얼마나 중요한지 깨닫게 되었다.
이번 프로젝트를 통해 단순한 화면 구현이 아닌, **팀 단위 개발의 흐름**을 경험할 수 있었다는 점이 가장 큰 수확이었다.
<br/><br/>

**한고은**<br/>
 **깃과 깃허브**를 그저 버전 관리용으로만 알고있었는데, 직접 다뤄보니 **생각보다 복잡하고 할 게 많았다.** 
처음엔 오류나는 것이 두려워 최대한 각자의 영역이 겹치지 않게 내 할 것만 잘 해야지… 무사히 지나갔으면 생각했는데, 다른 팀 발표를 들으며 지금 **오류와 충돌을 많이 내보며 익숙해지고 그 문제들을 해결해보는 것이 낫다**는 걸 깨달았다. 그래서 이후엔 자잘한 이슈들이 발생해도 겁먹지 않았던 것 같다. 찬찬히 뜯어보니 크게 어려울 거 없는 것들이라 이젠 비슷한 문제들이 발생해도 잘 대처할 수 있다.
집중해서 코드를 짜다보면 작은 단위로 add, commit 하기 어려워지는데 하나의 문제나 요구사항을 해결하면 의식적으로 바로바로 하려 노력했다. 확실히 브랜치를 따로 사용하니 코드 수정의 부담이 적었다. 커밋 메시지도 컨벤션을 따르려다 보니 익숙치 않아 계속 찾아보며 작성했는데 이젠 그것도 손에 익었다.
팀원들과 함께 **git과 github**를 사용하며 이 두 가지가 팀 단위 **프로젝트의 작업 효율을 높여주는 것**을 느꼈고, 한 달도 채 되지 않은 이 시점에 벌써 팀원들과 친해진 것 같다. 함께 작업하며 각자의 페이지에서 최선을 다하고, 문제가 발생했을 때 독려하고 해결하고자 나서준 팀원들에게 감사하다.
<br/><br/>

**한희나**<br/>
 우선 저번 초미니 프로젝트를 하면서 pull과 push에 대해 정말 몸으로 해보는게 얼마나 중요한지 깨달았다면 이번 프로젝트는 **협업의 중요도**에 대해 그리고 하나의 프로젝트를 완성하는 그 시간동안 해야하는 것에 대해 많이 알 수 있는 시간이였다.
또한 **개인 브랜치와 dev 브랜치 사용이 협업에서 얼마나 효율적**인지 알 수 있어 좋았다.
